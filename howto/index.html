Character Sheet System

How to add characters, edit data, and publish changes — for people who've never done this before.

Overview

This system is a set of static web pages hosted on GitHub Pages. Each character has their own folder containing three files: a stub HTML page, a JSON data file, and a portrait image. A shared JavaScript file reads the JSON and builds the sheet automatically — so you never touch the HTML to update a character.

The only files you'll ever edit for a character are character.json and occasionally roster.json when adding someone new.

Prerequisites
1

A GitHub Account

Free at github.com. You'll need one to clone the repo and push changes.

2

Git

Download from git-scm.com. After installing, open a terminal and confirm with git --version. Then set your identity once:

git config --global user.name "Your Name"
git config --global user.email "you@example.com"
3

Visual Studio Code (recommended)

Free at code.visualstudio.com. Has Git built in so you can commit and push without the terminal.

Clone the Repo

Cloning downloads a local copy of the repo that stays connected to GitHub so you can push changes back up.

1

In VS Code

Open the Command Palette (Ctrl+Shift+P), type Git: Clone, and paste the repo URL. Choose a local folder. VS Code will ask to open it — say yes.

2

Trust the Workspace

VS Code will ask if you trust the folder. Click Yes, I trust the authors. Without this the terminal and Git features won't work.

3

Authenticate with GitHub

The first time you push, VS Code opens a browser to authenticate. Follow the prompts — you only need to do this once per machine.

Rule of thumb: files inside a character folder are character-specific. Files at the root are shared. You'll mostly touch files inside character folders and roster.json.
Create the Character Folder

Each character needs their own folder. The folder name becomes part of the URL — keep it lowercase with no spaces (e.g. thorvald, not Thorvald Ironbeard).

1

Create the folder

In VS Code's file explorer, right-click the root of the project and choose New Folder. Name it after your character.

2

Add three files

Inside the folder you'll need: index.html, character.json, and portrait.png. The next three sections cover each one.

The index.html Stub

Copy an existing character's index.html into your new folder. Then update only these five lines — the OG meta tags that control how the page looks when shared on Discord:

<meta property="og:title"       content="CHARACTER NAME" />
<meta property="og:description" content="Race · Class · Level — Quote" />
<meta property="og:type"        content="website" />
<meta property="og:url"         content="https://USERNAME.github.io/sheets/yourcharacter/" />
<meta property="og:image"       content="https://USERNAME.github.io/sheets/yourcharacter/portrait.png" />
The paths ../sheet.css and ../sheet.js in the file use ../ because the stub is one folder deep. Don't change these.
Portrait Image

Save your character's portrait as portrait.png inside their folder. The sheet displays it up to 420px wide; the roster card crops it to a 120px thumbnail from the top center.

Tip: For best results on the roster card, use a portrait where the character's face is near the top of the image.
Update roster.json

Add one entry per character to roster.json. The primary and accent colors should match the character's theme and control the card's border glow and quote color.

[
  {
    "name":     "Thorvald",
    "subtitle": "Dwarf · Fighter · Level 4",
    "quote":    "\"I've survived worse.\"",
    "portrait": "./thorvald/portrait.png",
    "href":     "./thorvald/",
    "primary":  "#c8a060",
    "accent":   "#4a9aff"
  }
]
character.json — meta

Controls the page title and social embed previews (Discord, Slack, etc.).

FieldTypeDescription
titlestringBrowser tab title. E.g. "NIX — Character Sheet"
ogTitlestringEmbed title shown on Discord etc.
ogDescriptionstringEmbed subtitle. Typically Race · Class · Level — Quote.
ogUrlstringFull URL to the character page.
ogImagestringFull URL to the portrait. Used as embed thumbnail.
character.json — theme

Controls every color on the sheet. All values are hex codes. primary and accent are the most important — they define the character's visual identity.

FieldDescription
primaryMain color — names, stat highlights, section titles.
accentSecondary color — warnings, quotes, emotional weight.
_primary / _accentNotes to yourself about what each color means. Never rendered.
shadow / deep / surface / raisedBackground layers. Adjust for a warmer or cooler dark tone.
border / muted / text / textDimUI chrome. Safe to leave as defaults.
Tip: Think about what the two colors mean for your character, not just how they look. Intentionality carries through the whole sheet.
character.json — header
"header": {
  "name":     "NIX",        // large title, usually all-caps
  "subtitle": "Fetchling · Barbarian // Alchemist · Gestalt Level 3",
  "quote":    "\"This is not your party, Ember.\""
}
character.json — identity

Label/value pairs shown in a grid. Add as many as you need. The optional note renders dimmed in parentheses after the value.

"identity": [
  { "label": "Race",      "value": "Fetchling" },
  { "label": "True Name", "value": "Nix", "note": "formerly: Ember" }
]
character.json — appearance

Physical description items. labelVariant controls the label color.

"appearance": [
  { "label": "Build",   "value": "Lean.",               "labelVariant": "primary" },
  { "label": "Warning", "value": "Goes still before violence.", "labelVariant": "accent"  }
]

labelVariant: primary accent dim

character.json — abilityScores

Enter the raw score only — modifiers are calculated automatically using floor((score - 10) / 2). Negative mods display in accent color. Use misc for racial or item adjustments; they show in a hover tooltip.

"abilityScores": [
  { "name": "STR", "score": 18, "primary": true  }, // primary: true = highlighted border
  { "name": "DEX", "score": 17, "primary": true,
    "misc": 2, "miscReason": "Racial +2" }, // tooltip on hover
  { "name": "WIS", "score":  8, "primary": false }  // negative mod = accent color auto
]
character.json — combat

Provide the raw components — initiative, AC, saves, and attack bonuses are all calculated automatically.

"combat": {
  "bab":         3,         // Base Attack Bonus
  "armorBonus":  4,         // armor's AC contribution
  "armorMaxDex": 4,         // max DEX bonus (omit if no cap)
  // hp as per-level rolls — max is calculated automatically (roll + CON mod each level)
  "hp": [
    { "level": 1, "roll": "12", "class": "Barbarian" },
    { "level": 2, "roll": "7",  "class": "Barbarian",
      "misc": 1, "miscReason": "Toughness" }, // optional flat bonus + tooltip
    { "level": 3, "roll": "12", "class": "Barbarian" }
  ],
  // Current HP is tracked in-session via +/− controls on the HP tile.
  // Hovering the tile shows the per-level breakdown. "↺" resets to full.
  // Alternatively, pass a plain string for a static display: "hp": "45"
  "speed":       "30 ft",

  "saves": [
    { "label": "Fort", "stat": "CON", "base": 3 },
    { "label": "Will",  "stat": "WIS", "base": 1,
      "variant": "accent",                  // accent color (e.g. weak save)
      "misc": 2, "miscReason": "Iron Will"  // bonus + tooltip
    }
  ],

  "extraStats": [  // any extra tiles — class resources, pools, etc.
    { "label": "Rage Rounds", "value": "10" }
  ],

  "weapons": [
    {
      "name":         "Kukri",
      "stat":         "STR",      // governs attack bonus (STR or DEX)
      "damage":       "1d4+4",
      "crit":         "18-20 x2",
      "notes":        "Rule of Cool.",
      "notesVariant": "dim",     // "dim", "primary", or "accent"
      "misc":         1,         // optional attack bonus (feats, magic)
      "miscReason":   "Masterwork"
    }
  ]
}
character.json — classFeatures

One entry per class. For gestalt characters, use two entries — they render side by side. Descriptions support **bold** and *italic*.

"classFeatures": [
  {
    "className": "Fighter",
    "features": [
      { "name": "Bravery",       "description": "+1 Will saves vs. fear" },
      { "name": "Armor Training",  "description": "" }
    ]
  }
]
character.json — feats
"feats": [
  { "level": "1st", "name": "Power Attack", "notes": "-1 attack / +2 damage" }
]

Use "—" for level if the feat wasn't gained at a specific level.

character.json — skills

Only list skills the character has invested in or has a notable bonus on. Totals are calculated automatically. A ● indicator appears when the +3 class skill bonus is active.

Formula: stat mod + ranks + class bonus (+3 if classSkill: true and ranks > 0) + misc

"skills": [
  {
    "name":        "Perception",
    "stat":        "WIS",         // governing ability score
    "classSkill":  true,          // +3 bonus if ranks > 0
    "ranks":       5,
    "misc":        2,             // optional bonus
    "miscReason":  "Racial bonus" // shown in hover tooltip
  }
]
character.json — racialTraits
"racialTraits": {
  "raceName": "Fetchling",
  "traits": [
    { "name": "Darkvision",     "description": "60 ft" },
    { "name": "Shadow Blending", "description": "Miss chance in dim light increases to 50%" }
  ]
}
character.json — equipment
"equipment": [
  { "name": "Chain Shirt",     "notes": "+4 AC",           "notesVariant": "dim"     },
  { "name": "Health Potion",   "notes": "Emergency only.", "notesVariant": "accent"  },
  { "name": "Gold",             "notes": "150 gp",           "notesVariant": "primary" }
]

notesVariant: dim standard · accent warnings/voice · primary highlighted values

character.json — magic

Optional. Add a magic object to enable the formulae/spell section on the sheet. If omitted, the section is silently skipped — safe to leave out for non-casters.

The sheet displays slots per day, a prepared list (each entry tappable for full details), and an "Open Formula Book" button that shows all known spells grouped by level.

FieldTypeDescription
typestringClass type. Reserved for future rendering differences. Use "formulae" for Alchemist, "arcane" for Wizard/Magus, "spontaneous" for Sorcerer/Oracle, "divine" for Cleric/Druid.
castingModestringHow spell usage is tracked. "consume" (default): each prepared entry is a discrete instance — using it removes it from the list and the slot returns. "pool": prepared spells are a list; casting decrements a shared slot pool without removing the spell.
labelstringSection heading shown on the sheet. E.g. "Formulae" or "Spells".
spellbookLabelstringLabel for the book button and modal title. Defaults to "Spellbook" if omitted.
slotsPerDayobjectKeys are spell levels as strings ("1", "2", etc.). Values are total slots including ability score bonus slots — calculate this externally and store the final number.
slotsUsedobjectTracks slots spent this session. Same shape as slotsPerDay. Updated by the sheet during play; persist it between sessions via Export State. Start each day with {}.
preparedarrayArray of id strings matching entries in known. Update between sessions. Duplicates allowed (e.g. preparing the same formula twice). Use [] if nothing is prepared.
preparedTimestampnumberUnix timestamp (ms) of the last preparation change. Set automatically by the sheet. Used to detect stale state. You don't need to set this manually — Export State handles it.
knownarrayArray of spell/formula objects. This is the full formula book or spellbook. Use [] if not yet populated.

Each entry in known:

FieldTypeDescription
idstringUnique slug used to cross-reference prepared. Kebab-case recommended — e.g. "cure-light-wounds".
namestringDisplay name.
levelnumberSpell or extract level as an integer.
schoolstringOptional. Spell school, e.g. "Conjuration".
castingTimestringOptional. E.g. "1 standard action".
rangestringOptional. E.g. "Touch", "Close (25 ft + 5 ft/2 levels)".
targetstringOptional. E.g. "Creature touched".
durationstringOptional. E.g. "1 min/level", "Instantaneous".
savingThrowstringOptional. E.g. "Will negates", "Reflex half".
descriptionstringOptional. Supports **bold** and *italic*. Use \n\n for paragraph breaks.
 
"magic": {
  "type":           "formulae",
  "label":          "Formulae",
  "spellbookLabel": "Formula Book",
  "slotsPerDay": {
    "1": 4            // base slots + INT/CHA bonus combined
  },
  "prepared": ["enlarge-person", "cure-light-wounds"], // update each session
  "known": [
    {
      "id":          "cure-light-wounds",
      "name":        "Cure Light Wounds",
      "level":       1,
      "school":      "Conjuration",
      "castingTime": "1 standard action",
      "range":       "Touch",
      "target":      "Creature touched",
      "duration":    "Instantaneous",
      "savingThrow": "Will half (harmless)",
      "description": "Heals 1d8 + caster level hit points (max +5)."
    }
  ]
}
Alchemist note: Alchemists do not have access to all formulae automatically. They start with 2 + INT modifier known formulae, gain 1 per level, and can add more like a wizard adds spells. The prepared array represents extracts mixed for the day — duplicate entries are valid if the same formula is prepared more than once.
Slots per day includes the ability score bonus. For an Alchemist with INT 16 (+3) at level 3: 3 base + 1 INT bonus = 4. Calculate it yourself and store the final total.
Managing spells during a session

Each prepared spell has two buttons:

  • Use / Cast — marks the spell as spent. In consume mode this removes one instance from the prepared list (the slot effectively returns). In pool mode this decrements the shared slot count without removing the spell from the list.
  • ✕ (Unprepare) — removes the spell from the prepared list without spending a slot. Use this to undo a preparation mistake or swap spells mid-day with GM permission.

Spells can also be prepared directly from the spellbook modal. In pool mode, the spellbook's Prepare button toggles — if the spell is already prepared it shows "Unprepare" instead.

Saving state between sessions

The sheet tracks current HP and spell usage in your browser's local storage while you play. To persist that state into character.json so it survives a browser reset or carries between devices:

1
Click "Export State" in the top-right corner of the sheet header. A modal opens with a JSON snippet containing currentHp and a magic block with slotsUsed, prepared, and preparedTimestamp.
2
Click "Copy" in the modal to copy the JSON to your clipboard.
3
Open character.json and find the magic object. Replace (or add) the slotsUsed, prepared, and preparedTimestamp fields with the values from the exported magic block. The currentHp value is for reference — current HP is stored in local storage only and does not go back into the JSON.
4
Save and commit. The sheet will load this state on next open.
At the start of a new day, reset slotsUsed to {} and replace prepared with the new day's spell list, then commit before the session.
character.json — lore

A list of blocks making up the backstory section. Each has a title, border accent color, and one or more content types. Blocks render in order: content → bindings → incidents → hooks.

FieldTypeDescription
titlestringBlock heading.
titleNotestringOptional dimmed note after the title. E.g. "(she/her)"
accent"primary" | "accent"Left border color. Alternate for visual rhythm.
contentstringProse. Supports **bold**, *italic*. Use \n\n for paragraph breaks. Wrap paragraphs in a blockquote box using > to open and \n>\n\n to close.
incidentsarrayQuoted lines in left-border style. Each has text and variant ("default" or "accent").
bindingsarrayNamed ability/curse blocks. Each has label, name, description.
hooksarrayCampaign hook strings, rendered as a bulleted list.
 
"lore": [
  {
    "title":   "The Beginning",
    "accent":  "primary",
    "content": "First paragraph.\n\nSecond paragraph. **Bold** and *italic* work."
  },
  {
    "title":      "Things They Say",
    "accent":     "accent",
    "incidents": [
      { "text": "\"I've survived worse.\"", "variant": "default" },
      { "text": "\"...barely.\"",            "variant": "accent"  }
    ]
  },
  {
    "title":  "Campaign Hooks",
    "accent": "primary",
    "hooks":  ["Why did they leave?", "What does the villain want?"]
  }
]
Extra Lore Pages — Overview

Characters can have additional lore pages beyond what fits in the main character sheet — extended backstory pieces, in-character vignettes, session recaps, anything you want to give a proper home.

These live in a lore/ subfolder inside the character's folder. The character sheet automatically detects the folder and renders a Further Reading section at the bottom of the backstory area with links to each entry. No changes needed to character.json.

Each lore page loads the character's color theme automatically, has a nav bar showing all lore entries, and a back link to the main sheet.

Zero config: just create the lore/ folder with the right files and the sheet picks it up. Remove the folder and the section disappears.
Extra Lore Pages — Folder Structure

Inside your character's folder, create a lore/ subfolder with three things:

yourcharacter/lore/
index.html — copy from an existing character's lore folder, never edit
index.json — list of lore entries (controls the nav and Further Reading section)
entry-name.json — one file per lore entry
another-entry.json — add as many as you want

The lore page URL uses a hash to select which entry to display — e.g. sheets/nix/lore/#nix-combat. Changing the hash navigates between entries without a page reload.

Extra Lore Pages — lore/index.json

A simple array — one entry per lore page. This controls both the Further Reading section on the character sheet and the nav bar on the lore pages.

[
  {
    "id":          "nix-combat",        // matches the filename (without .json)
    "title":       "Nix in Combat",     // shown in nav and Further Reading
    "description": "The thing about Nix is the constant hum of activity.",
    // optional — shown as a subtitle in Further Reading
    "shareUrl":    "./lore/nix-combat/"  // relative to character root, not lore folder
  },
  {
    "id":    "the-docks",
    "title": "The Night at the Docks"
  }
]

The id must match the filename exactly — "nix-combat" maps to nix-combat.json.

The optional shareUrl field points to the Discord embed stub folder (see Discord Embeds below). When present, the Further Reading link and the lore page nav both use it instead of the hash URL.

Extra Lore Pages — Lore Entry JSON

Each lore entry is its own JSON file. It supports two formats — plain prose for narrative pieces, or the same block layout used in the main character sheet for more structured content.

Plain prose — best for vignettes, fiction, in-character writing:

{
  "title":   "Nix in Combat",
  "content": "First paragraph.

Second paragraph.

*Italic text.* **Bold text.**"
}

Use for paragraph breaks. Supports **bold** and *italic*.

Block layout — best for structured lore, timelines, or content that benefits from the raised block styling:

{
  "title": "The Docks — Session 4",
  "blocks": [
    {
      "title":   "Before",
      "accent":  "primary",
      "content": "What happened before the session."
    },
    {
      "title":      "What Was Said",
      "accent":     "accent",
      "incidents": [
        { "text": ""SHUT UP EMBER."", "variant": "default" }
      ]
    }
  ]
}

The blocks array uses the same schema as the lore array in character.json — same field names, same content types (content, incidents, bindings, hooks). Refer to the lore schema section above for full details.

You can mix both formats in one file — use content for an intro paragraph and blocks for structured sections below it. They render in that order.
Extra Lore Pages — Discord Embeds

Social platforms like Discord read OG meta tags before JavaScript runs, so the hash-based lore URLs (lore/#nix-combat) all look identical to an embed crawler — it can't see the hash. The fix is a small stub folder per lore entry containing a plain HTML file with hardcoded OG tags that immediately redirects to the real lore page.

It's a bit of a workaround, but it works cleanly. The stub is tiny and you only need one per entry you want to be shareable.

Create a folder inside lore/ matching the entry id, with a single index.html:

yourcharacter/lore/
index.json
nix-combat.json
nix-combat/ — one folder per shareable entry
index.html — OG stub + redirect

The stub index.html contains only OG meta tags and a redirect — copy this template and update the five highlighted values:

<!DOCTYPE html>
<html lang="en">
<head>
<meta charset="UTF-8">
<meta property="og:title"       content="Nix in Combat" />
<meta property="og:description" content="The thing about Nix is the constant hum of activity." />
<meta property="og:type"        content="website" />
<meta property="og:url"         content="https://USERNAME.github.io/sheets/nix/lore/nix-combat/" />
<meta property="og:image"       content="https://USERNAME.github.io/sheets/nix/portrait.png" />
<title>Nix in Combat</title>
<meta http-equiv="refresh" content="0;url=../#nix-combat">
</head>
<body>
  <script>window.location.replace('../#nix-combat');</script>
</body>
</html>

The meta http-equiv="refresh" handles browsers with JavaScript disabled; the script handles everyone else. Both redirect to the hash URL where the actual lore page renders.

Once the stub folder exists, add "shareUrl": "./lore/nix-combat/" to the entry in lore/index.json — the Further Reading link and lore page nav will automatically use the stub URL instead of the hash URL, so shared links always go through the stub and get the right embed.

Important: shareUrl is relative to the character sheet root (yourcharacter/), not the lore folder. Always prefix with ./lore/ — e.g. "./lore/nix-combat/", not "./nix-combat/".
The stub is optional per entry — entries without a shareUrl still work fine, they just won't have a custom Discord embed.
Commit & Push

Once you've made changes, commit them (save a snapshot) and push (send to GitHub).

1

Open Source Control

Click the branch icon in VS Code's left sidebar, or press Ctrl+Shift+G.

2

Stage your changes

Hover a changed file and click +, or click Stage All Changes.

3

Write a commit message

Type a brief description — e.g. Add Thorvald character or Update Nix HP. Click Commit.

4

Push

Click Sync Changes in the Source Control panel. This sends your commit to GitHub.

GitHub Pages

After pushing, GitHub automatically rebuilds and deploys the site — usually 30–60 seconds. Watch progress under the Actions tab in your repo. A green checkmark means the deploy is live.

First time setup: Go to your repo on GitHub → Settings → Pages → Source: Deploy from a branch → Branch: main → Folder: / (root) → Save. Only needed once.

Your roster will be live at https://YOUR-USERNAME.github.io/sheets/ and each character at https://YOUR-USERNAME.github.io/sheets/charactername/.
